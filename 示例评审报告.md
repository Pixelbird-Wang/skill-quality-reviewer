# 示例评审报告

本文演示如何使用 `skill-quality-reviewer` 评审一个已有 skill。

这里的目标不是证明某个 skill "好"或"坏"，
而是展示：

- 这套标准如何落地
- 评审结果应该怎样写
- 什么叫结构化结论

## 评审对象

- Skill：`context-engineering-advisor`
- 类型：Interactive Skill
- 场景：用于诊断 AI 工作流中的 context stuffing 与 context engineering 问题

## 一、总结结论

- 总分：`82/100`
- 评级：`良好，可上线`
- 是否可上线：`可上线`

结论说明：

这个 skill 是一个整体质量较高的诊断型 skill。

它的核心问题定义明确，交互引导能力强，教学性也比较好，
因此很适合 PM 在复杂 AI 工作流场景下使用。

但它也存在两个典型问题：

1. 主文件偏长，上下文成本略高
2. 完成判定和验证机制偏软，更多依赖引导而不是硬闭环

因此它是一个"好 skill"，但不是一个"满分 skill"。

## 二、P0 问题

未发现 P0 阻断问题。

说明：

- 触发边界基本清楚
- 主体内容可执行
- 没有明显危险动作
- 没有本质失控的 scope 设计

## 三、十维度评分

- 触发质量：4/5
- 边界控制：4/5
- 执行性：4/5
- 结构质量：4/5
- 工具/接口清晰度：3/5
- 验证与完成判定：3/5
- 效率与上下文成本：3/5
- 安全与误用防护：4/5
- 可组合性与复用性：4/5
- 可维护性：4/5

## 四、主要优点

### 1. 问题定义非常清楚

它不是泛泛地讨论"如何写 prompt"，
而是把问题准确框定为：

- context stuffing
- context engineering

这使它的触发场景很明确，也减少了误用概率。

### 2. 教学性强

它不仅能给 agent 用，
也能帮助 PM 理解为什么当前 AI 工作流会失控。

这类 skill 很适合"边用边学"的场景。

### 3. 交互引导完整

它不是一句话建议型 skill，
而是有明确的上下文采集、症状识别、诊断问题和后续行动建议。

对复杂问题来说，这是加分项。

### 4. 可拆出结构化诊断信号

它引入了多个明确的诊断问题，
这意味着用户的模糊痛点能被转成结构化问题。

这类 skill 在组织治理中很有价值。

## 五、主要问题

### 1. 主文件篇幅偏长

这是它最明显的问题。

主文件里承载了大量解释性内容、理论说明和引导信息，
虽然这提升了教学性，
但也提高了上下文成本。

风险在于：

- agent 的注意力被摊薄
- 主路径与背景说明耦合较重
- 后续维护时更容易局部失焦

### 2. 完成判定偏软

它的强项是"诊断引导"，
不是"强闭环执行"。

所以它在 done condition 上不够硬。

例如：

- 什么情况下诊断算完成
- 输出到什么程度就该停止
- 是否已经形成最小行动方案

这些点仍然可以更明确。

### 3. 验证机制偏弱

它能帮助用户看到问题，
但对"是否已经成功完成诊断"没有非常强的验证动作。

这类问题不会让它失效，
但会让不同轮次之间的输出一致性变差。

### 4. 工具接口层还不够明确

如果未来要进一步接 memory、RAG、知识库、或检索类工具，
当前 skill 中关于工具边界和接口选择的规范还不够强。

现在它更像一个高质量认知诊断 skill，
而不是一个强工具编排 skill。

## 六、命中的反模式

### 1. 轻度上下文膨胀

它没有严重膨胀，
但已经出现了典型迹象：

- 主文件中解释内容偏多
- 理论与关键路径耦合
- 不是所有内容都需要在首轮加载

这属于"轻度命中"，不是致命问题。

## 七、最小修改建议

### 1. 进一步外置长篇理论说明

把更多背景性、解释性内容放到参考文件，
主 `SKILL.md` 聚焦在：

- 什么时候用
- 先问什么
- 怎么判断问题
- 产出什么

### 2. 增加明确结束条件

建议增加类似这样的 done definition：

- 已识别出主要症状
- 已定位至少一个关键边界问题
- 已输出三条行动建议
- 已给出下一轮优化重点

满足这些后应明确结束。

### 3. 补充固定输出模板

例如固定输出：

1. 主要症状
2. 根因判断
3. 上下文边界问题
4. 立即行动建议
5. 后续优化建议

这样可以提高不同会话之间的一致性。

## 八、这个示例说明了什么

这个例子说明了三件事：

### 1. 好 skill 不等于所有维度都满分

一个 skill 只要：

- 触发清楚
- 执行可靠
- 边界合理
- 风险可控

即使上下文成本略高、验证略软，仍然可能是一个"好 skill"。

### 2. 评审不是挑刺，而是定位最小改进点

评审的重点不是证明作者写得不行，
而是找出：

- 最大收益的问题点
- 最小可行修改路径

### 3. "长且全"不能自动加分

这个示例特别适合帮助团队建立共识：

**教学性强、内容丰富是优点，但不能替代触发、执行、验证和效率。**

## 九、可复用模板

如果你要用这份示例去评审别的 skill，可以直接替换以下内容：

- Skill 名称
- 评审对象简介
- 十维度评分
- 优点
- 问题
- 反模式
- 最小修改建议

建议始终保留这套结构：

1. 总结结论
2. P0 问题
3. 十维度评分
4. 主要优点
5. 主要问题
6. 命中的反模式
7. 最小修改建议

这能保证不同评审者之间的输出风格一致。
