# 示例评审报告

本文演示如何使用 `skill-quality-reviewer` 评审一个已有 skill。

这里的目标不是证明某个 skill "好"或"坏"，
而是展示：

- 这套标准如何落地
- 评审结果应该怎样写
- 什么叫结构化结论
- 如何体现 Skill 类型识别和三层护栏判断

## 评审对象

- Skill：`context-engineering-advisor`
- 类型：Interactive Skill
- 场景：用于诊断 AI 工作流中的 context stuffing 与 context engineering 问题

## 一、总结结论

- 总分：`82/100`
- 评级：`良好，可上线`
- 是否可上线：`可上线`

结论说明：

这个 skill 是一个整体质量较高的诊断型 skill。

它的核心问题定义明确，交互引导能力强，教学性也比较好，
因此很适合 PM 在复杂 AI 工作流场景下使用。

但它也存在两个典型问题：

1. 主文件偏长，上下文成本略高
2. 完成判定和验证机制偏软，更多依赖引导而不是硬闭环

因此它是一个"好 skill"，但不是一个"满分 skill"。

## 二、Skill 类型识别

- 设计模式：**思维框架型**（核心目的是引导用户进行结构化诊断，而非执行具体操作步骤）
- 执行模式：**交互式**（依赖用户提供上下文信息，核心步骤有"询问用户"操作）
- 评审侧重说明：作为思维框架型 skill，重点关注"量化阈值是否明确""非目标约束是否清楚"；作为交互式 skill，完成条件允许依赖用户判断，不触发自主模式的额外严格要求。

## 三、P0 问题

未发现 P0 阻断问题。

说明：

- 触发边界基本清楚，description 有基本触发场景描述
- 主体内容可执行，有诊断引导步骤
- 没有高风险动作，安全防护要求低
- 没有本质失控的 scope 设计
- 执行模式为交互式，自主模式 P0 不适用

## 五、主要优点

### 1. 问题定义非常清楚

它不是泛泛地讨论"如何写 prompt"，
而是把问题准确框定为：

- context stuffing
- context engineering

这使它的触发场景很明确，也减少了误用概率。

### 2. 交互引导完整

它不是一句话建议型 skill，
而是有明确的上下文采集、症状识别、诊断问题和后续行动建议。

对复杂问题来说，这是加分项。

### 3. 可拆出结构化诊断信号

它引入了多个明确的诊断问题，
这意味着用户的模糊痛点能被转成结构化问题。

这类 skill 在组织治理中很有价值。

### 4. 安全风险低，三层护栏要求自然满足

作为纯诊断型 skill，本身不执行危险操作，
安全层面的自然防护使其整体风险可控。

## 六、主要问题

### 1. description 缺少触发短语和时序说明

当前 description 更多是功能描述，
而非用户可能说出的触发短语。

例如缺少：
- "当你的 AI 工作流输出不稳定时使用"
- "在优化 prompt 之前使用"

这增加了路由不准的风险。

### 2. 主文件篇幅偏长

主文件里承载了大量解释性内容和理论说明，
虽然提升了教学性，
但教学内容占比超过 40%，超出推荐阈值。

风险在于：
- agent 的注意力被摊薄
- 主路径与背景说明耦合较重
- 后续维护时更容易局部失焦

### 3. 完成判定偏软，缺少可观测标准

所有完成条件都是定性描述，
没有明确的"做完的标志"。

例如缺少：
- 已识别出至少 2 个主要症状
- 已输出至少 3 条行动建议
- 已给出下一轮优化的首要重点

### 4. 缺少量化阈值和负面指令

作为思维框架型 skill，没有量化分析深度要求，
也没有明确的"不要做 X"负面指令，
容易导致 agent 停留在浅层诊断就提前结束。

## 七、命中的反模式

### 1. 轻度上下文膨胀（命中反模式 6）

严重程度：轻度

主文件中解释内容偏多，理论与关键路径耦合，
教学性内容超过 40%。

这属于"轻度命中"，不是致命问题，但影响效率评分。

## 八、最小修改建议

### 1. 优化 description 加入触发短语和时序位置（优先级：高）

**改什么：** `description` 字段

**怎么改：** 在现有描述基础上补充：
- 2-3 个用户可能说的触发短语（"AI 输出不稳定时"/"优化 prompt 之前"）
- 时序说明（"在开始优化工作流之前使用"）
- 负向条件（"不适用于单一 prompt 调试场景"）

**改后效果：** 路由准确率提升，减少误触发。

### 2. 外置长篇理论说明（优先级：中）

**改什么：** 主 `SKILL.md` 中的教学性/解释性内容

**怎么改：** 把背景理论、概念解释移入参考文件，
主 `SKILL.md` 聚焦在：
- 什么时候用
- 先问什么
- 怎么判断问题
- 产出什么

**改后效果：** 主文件 token 降至合理区间，关键路径更清晰。

### 3. 增加明确完成条件（优先级：中）

**改什么：** 在 Phase 最后增加 done definition

**怎么改：** 添加类似：
- 已识别至少 2 个主要症状 ✓
- 已定位至少 1 个关键边界问题 ✓
- 已输出 3 条可执行行动建议 ✓
- 已给出下一轮优化重点 ✓

**改后效果：** 不同会话间输出一致性提高。

### 4. 补充量化阈值和负面指令（优先级：低）

**改什么：** 诊断步骤

**怎么改：**
- 增加"至少识别 3 个维度的症状"这类硬性要求
- 增加"不要在诊断完成前给出优化建议"这类负面指令

**改后效果：** 防止 agent 浅尝辄止，提升诊断深度的一致性。

## 九、这个示例说明了什么

### 1. 类型识别改变了评审视角

识别出"思维框架型 + 交互式"后，
评审者就不会用"循环退出检查清单"或"机器可验证完成条件"来苛责它，
而是聚焦在思维工具的量化深度和非目标约束是否清楚。

**不识别类型就套用统一标准，会导致评审结论失准。**

### 2. 安全维度要区分风险等级

这个 skill 安全维度得 4 分，
不是因为三层护栏很完整，
而是因为它本身是低风险 skill，安全防护要求自然低。

高风险 skill 在安全维度必须三层均覆盖才能得 4 分以上。

### 3. 好 skill 不等于所有维度都满分

一个 skill 只要：
- 触发清楚
- 执行可靠
- 边界合理
- 风险可控

即使上下文成本略高、验证略软，仍然可能是一个"好 skill"。

### 4. 评审不是挑刺，而是定位最小改进点

评审的重点不是证明作者写得不行，
而是找出：
- 最大收益的问题点
- 最小可行修改路径

## 十、可复用模板

如需复用此报告结构评审其他 skill，直接使用 `SKILL.md` 中 Phase 6 的输出格式模板即可。

替换内容：Skill 名称、类型识别结论、评审对象简介、十维度评分、优点、问题、反模式、最小修改建议。
