## Purpose

用于评审另一个 skill 的质量，判断它是否属于“好的 skill”。

这里的“好”不是指文字多、看起来专业、像专家发言，
而是指这个 skill 是否真正能让 agent：

- 正确触发
- 正确执行
- 正确停止
- 正确验证
- 尽量少走弯路
- 尽量少带来安全和维护风险

一句话标准：

**好的 skill，不是“写得像专家”，而是“能让 agent 稳定做对事”。**

本 skill 适用于四类场景：

1. 新 skill 上线前评审
2. 存量 skill 质量体检
3. 多个 skill 候选方案横向比较
4. 组织内部建立 skill 质量门禁

## Key Concepts

### 一、什么是“好 skill”

一个好的 skill，必须同时满足以下八个要求：

1. **容易选中**：agent 知道什么时候该用它
2. **容易执行**：agent 知道具体该怎么做
3. **不容易误用**：边界和限制清楚
4. **成本可控**：不会把大量无关上下文硬塞进来
5. **可验证**：知道怎样判断有没有做对
6. **可组合**：能和别的 skill 一起工作
7. **可维护**：别人能改、能升级、能替换
8. **足够安全**：不会引导 agent 做危险动作而没有防护

### 二、什么不是“好 skill”

以下这些特征本身都不等于高质量：

- 文字很多
- 看起来很完整
- 很像真人专家说话
- 充满“最佳实践”“深度思考”“灵活判断”这类术语
- 带有人设、语气、偏好、风格，但没有可执行动作

### 三、评审的基本哲学

#### 1. 执行性优先于表述气质

只要不能转化为明确动作、判断点或验证动作，就不是强指令。

#### 2. 精准优先于“全面感”

很多 skill 失败，不是因为写少了，而是因为写太多、太泛、太想面面俱到。

#### 3. 分层优先于堆料

好的 skill 不会在核心文件里把所有知识一次性灌进上下文。

#### 4. 显性流程适合写成 skill，隐性判断不适合硬蒸馏

skill 擅长承载：

- 明确流程
- IF-THEN 规则
- 结构化检查清单
- 工具使用规范
- 产物模板
- 可验证步骤

skill 不擅长承载：

- 审美与品味
- 高维权衡
- 模糊直觉
- 组织政治判断
- “像某个资深专家那样想问题”

### 四、十个评审维度

1. 触发质量
2. 边界控制
3. 执行性
4. 结构质量
5. 工具/接口清晰度
6. 验证与完成判定
7. 效率与上下文成本
8. 安全与误用防护
9. 可组合性与复用性
10. 可维护性

### 五、典型反模式

1. 人设伪装能力
2. 全面但不可执行
3. 工具描述歧义
4. 工作流蔓延
5. 无验证真空
6. 上下文膨胀
7. 样例绑架
8. 硬编码隐性判断

## Application

### Phase 0：获取评审对象

最低需要：

- `name`
- `description`
- 核心正文

最好还能拿到：

- skill 目录结构
- 引用的参考文件
- 相关脚本
- 真实使用场景或失败案例

### Phase 1：触发层审查

检查：

1. `name` 是否具体、可检索、低歧义
2. `description` 是否同时说清做什么、什么时候用
3. 是否容易与相邻 skill 混淆
4. 是否存在明显误触发或漏触发风险

### Phase 2：边界与 scope 审查

检查：

1. 是否只解决一个连贯主问题
2. 是否说明不做什么
3. 是否混入多个独立职责
4. 是否试图蒸馏完整岗位/完整人设/完整思维方式

### Phase 3：执行性审查

检查：

1. 是否有清晰步骤
2. 是否有动作动词
3. 是否有分支判断
4. 是否有输入要求
5. 是否有输出要求
6. 是否有停止条件

### Phase 4：结构与效率审查

检查：

1. 主 `SKILL.md` 是否聚焦
2. 是否把次级内容放入参考文件
3. 是否有重复
4. 是否区分关键路径和可选路径
5. 是否有明显上下文膨胀

### Phase 5：验证、安全、组合性审查

#### A. 验证与完成判定

检查：

1. 是否定义何时停止
2. 是否定义如何判断成功
3. 是否要求验证后再宣布完成

#### B. 安全与误用防护

检查：

1. 是否涉及危险动作（删除、写入、发送通知等）
2. 是否涉及网络、生产、密钥、外部系统调用
3. 是否有足够 guardrail
4. **高风险 skill 特别检查**：若被评审 skill 涉及密钥管理、生产环境操作、用户数据写入，需在评分时对"安全与误用防护"维度至少评为 2 分以下（除非有完整防护），并标记为 P0 风险候选项

#### C. 可组合性

检查：

1. 是否有清晰职责边界
2. 能否作为其他 skill 的模块使用
3. 是否过度绑定个人或项目上下文

### Phase 6：输出结论

必须输出：

1. 总分与评级
2. 是否可上线
3. P0 问题
4. 十维度评分
5. 主要优点
6. 主要问题
7. 命中的反模式
8. 最小修改建议

#### 输出格式模板

```
# Skill 质量评审报告

## 一、总结结论
- 总分：__/100
- 评级：优秀/良好/可用待改/较弱/不合格
- 是否可上线：可上线/不可上线

结论说明：（2-3句话概括整体评价）

## 二、P0 问题
（列出所有P0阻断问题，如无则写"未发现P0阻断问题"并说明原因）

## 三、十维度评分
- 触发质量：__/5
- 边界控制：__/5
- 执行性：__/5
- 结构质量：__/5
- 工具/接口清晰度：__/5
- 验证与完成判定：__/5
- 效率与上下文成本：__/5
- 安全与误用防护：__/5
- 可组合性与复用性：__/5
- 可维护性：__/5

## 四、主要优点
（按影响程度降序列出，每条附简要说明）

## 五、主要问题
（按影响程度降序列出，每条附具体表现和改进方向）

## 六、命中的反模式
（列出命中的反模式名称，附具体表现和严重程度）
```

## Examples

### 好 skill 示例

一个用于评审 PRD 结构完整性的 skill，如果：

- 触发明确
- 边界清楚
- 有固定检查步骤
- 有输出模板
- 有完成和验证条件

那么它通常是一个好 skill。

### 坏 skill 示例

一个 skill 如果大量强调“顶级专家视角”“综合判断”“最佳实践”，
但没有输入、步骤、输出、完成条件和验证机制，
它通常不是一个好 skill。

## Common Pitfalls

- 把“像专家说话”误当成“能稳定执行”
- 把所有边界情况都塞进主文件，导致主路径失焦
- skill 名称和描述太泛，导致路由混乱
- 只写流程，不写完成条件
- 只写建议，不写验证
- 试图蒸馏一个岗位的全部经验，而不是一项能力
