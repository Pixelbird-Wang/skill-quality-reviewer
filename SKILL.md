## Purpose

用于评审另一个 skill 的质量，判断它是否属于"好的 skill"。

**安全声明：** 本 skill 仅对被评审 skill 的文本内容进行审查，不会执行、安装或运行被评审的 skill。

这里的"好"不是指文字多、看起来专业、像专家发言，
而是指这个 skill 是否真正能让 agent：

- 正确触发
- 正确执行
- 正确停止
- 正确验证
- 尽量少走弯路
- 尽量少带来安全和维护风险

一句话标准：

**好的 skill，不是"写得像专家"，而是"能让 agent 稳定做对事"。**

本 skill 适用于四类场景：

1. 新 skill 上线前评审
2. 存量 skill 质量体检
3. 多个 skill 候选方案横向比较
4. 组织内部建立 skill 质量门禁

## Key Concepts

### 一、什么是"好 skill"

一个好的 skill，必须同时满足以下八个要求：

1. **容易选中**：agent 知道什么时候该用它
2. **容易执行**：agent 知道具体该怎么做
3. **不容易误用**：边界和限制清楚
4. **成本可控**：不会把大量无关上下文硬塞进来
5. **可验证**：知道怎样判断有没有做对
6. **可组合**：能和别的 skill 一起工作
7. **可维护**：别人能改、能升级、能替换
8. **足够安全**：不会引导 agent 做危险动作而没有防护

### 二、什么不是"好 skill"

以下这些特征本身都不等于高质量：

- 文字很多
- 看起来很完整
- 很像真人专家说话
- 充满"最佳实践""深度思考""灵活判断"这类术语
- 带有人设、语气、偏好、风格，但没有可执行动作

### 三、评审的基本哲学

#### 1. 执行性优先于表述气质

只要不能转化为明确动作、判断点或验证动作，就不是强指令。

#### 2. 精准优先于"全面感"

很多 skill 失败，不是因为写少了，而是因为写太多、太泛、太想面面俱到。

#### 3. 分层优先于堆料

好的 skill 不会在核心文件里把所有知识一次性灌进上下文。
token 预算参考：Frontmatter ~100 tokens，主文件 2K-5K tokens，参考文档按需加载，总上下文占用 <10K tokens。

#### 4. 显性流程适合写成 skill，隐性判断不适合硬蒸馏

skill 擅长承载：

- 明确流程
- IF-THEN 规则
- 结构化检查清单
- 工具使用规范
- 产物模板
- 可验证步骤

skill 不擅长承载：

- 审美与品味
- 高维权衡
- 模糊直觉
- 组织政治判断
- "像某个资深专家那样想问题"

### 四、六种 Skill 设计模式

不同模式的 skill，评审侧重点不同。识别模式是评审的前提。

| 模式 | 适用场景 | 评审侧重 |
|------|----------|----------|
| **线性流程型** | 部署、安装、迁移等有明确步骤的操作 | 步骤完整性、安全默认值、降级方案 |
| **决策树型** | 大型平台选型、产品导航、问题诊断 | 分支导航清晰度、渐进式披露、主文件轻量化 |
| **循环迭代型** | TDD、代码审查、设计评审等反复执行的流程 | 退出条件、防偷懒机制、量化阈值 |
| **接力棒型** | 跨 session 长期项目，多 agent 协作 | 状态持久化机制、续命步骤防遗漏标注 |
| **多阶段编排型** | 复杂多周流程，有明确 Go/No-Go 检查点 | 检查点质量、子 skill 编排清晰度、时间成本透明 |
| **思维框架型** | 安全审计、架构分析等需要深度思考的场景 | 思维工具明确性、非目标约束、量化分析深度阈值 |

### 五、十个评审维度

1. 触发质量
2. 边界控制
3. 执行性
4. 结构质量
5. 工具/接口清晰度
6. 验证与完成判定
7. 效率与上下文成本
8. 安全与误用防护
9. 可组合性与复用性
10. 可维护性

### 六、典型反模式

1. 人设伪装能力
2. 全面但不可执行
3. 工具描述歧义
4. 工作流蔓延
5. 无验证真空
6. 上下文膨胀
7. 样例绑架
8. 硬编码隐性判断
9. 自主模式歧义（新）
10. 护栏层缺失（新）

## Application

### Phase 0：获取评审对象

最低需要：

- `name`
- `description`
- 核心正文

最好还能拿到：

- skill 目录结构
- 引用的参考文件
- 相关脚本
- 真实使用场景或失败案例

### Phase 0.5：识别 Skill 类型

**必须在评审正式开始前完成此步骤。** 类型识别决定后续各 Phase 的评审侧重。

#### 步骤一：识别设计模式

对照下方特征判断属于哪种模式（可以是复合模式）：

- **线性流程型**：有编号步骤，步骤间有顺序依赖，适合"先做 A 再做 B"描述
- **决策树型**：有显著分支结构，不同输入走不同路径，包含导航到子文件的指引
- **循环迭代型**：有明确的"做→验证→改进"循环，有退出检查清单
- **接力棒型**：有跨 session 状态文件机制，有续命步骤，适合长期/多 agent 场景
- **多阶段编排型**：分多个 Phase，每个 Phase 有独立 Activities/Outputs/Decision Point，调用其他 skill
- **思维框架型**：核心目的是控制 LLM 的分析方式而非执行步骤，有大量思维工具和非目标约束

#### 步骤二：识别执行模式

- **交互式（Interactive）**：依赖用户实时输入或确认，核心步骤中有"请用户"操作
- **自主式（Autonomous）**：整个流程无需用户干预即可完成
- **混合式（Hybrid）**：关键决策点有人工确认，其余步骤自主执行

#### 步骤三：调整评审侧重

根据识别结果，在后续 Phase 中对以下维度加强关注：

| 类型 | 加强关注的维度 |
|------|---------------|
| 线性流程型 | 安全默认值、步骤完整性、降级方案 |
| 决策树型 | 结构质量（分层）、触发质量（分支入口清晰度） |
| 循环迭代型 | 验证与完成判定（退出条件）、执行性（防偷懒机制） |
| 接力棒型 | 结构质量（状态文件机制）、验证与完成判定（续命步骤） |
| 多阶段编排型 | 边界控制（子 skill 职责分离）、验证与完成判定（检查点） |
| 思维框架型 | 执行性（量化阈值）、边界控制（非目标约束） |
| 自主式 | 验证与完成判定（机器可验证）、安全与误用防护（超时边界） |

### Phase 1：触发层审查

检查：

1. `name` 是否具体、可检索、低歧义
2. `description` 是否同时说清做什么、什么时候用
3. `description` 是否包含**触发短语**（用户可能说的话，而非纯功能描述）
4. 是否说明**时序位置**（"在 X 之前/之后使用"）
5. 是否包含**负向触发条件**（"不适用于 X 场景"）
6. 是否容易与相邻 skill 混淆（语义重叠是否过高）
7. 是否声明**执行模式适配**（仅交互式 / 仅自主式 / 两者均支持）
8. 是否存在明显误触发或漏触发风险

**判断基准：** 好的 description 应能回答"什么时候用我"和"什么时候不用我"两个问题。如果两者都只回答了一个，是扣分项。

### Phase 2：边界与 scope 审查

检查：

1. 是否只解决一个连贯主问题
2. 是否说明不做什么（明确排除范围）
3. 是否混入多个独立职责
4. 是否试图蒸馏完整岗位/完整人设/完整思维方式
5. 对于**多阶段编排型**，是否明确声明调用了哪些子 skill，职责边界是否清晰

### Phase 3：执行性审查

检查：

1. 是否有清晰步骤（编号、顺序、动作动词）
2. 是否有**量化阈值**而非定性描述（"至少3项""最多5步"而非"足够多的"）
3. 是否有**负面指令**（"不要做 X""禁止 Y"）明确列出
4. 是否有**强硬语气**防止 agent 偷懒（"必须""立即""不得跳过"）
5. 是否有**分支判断**（IF-THEN 规则，让 agent 知道不同情况走不同路径）
6. 是否有输入要求（输入格式、必需字段、前置条件）
7. 是否有输出要求（输出格式、必含字段、质量标准）
8. 是否有停止条件（done definition）
9. 对于**循环迭代型**，是否有明确的循环退出检查清单

**判断基准：** 对任意一条步骤问"具体怎么做"，如果无法在 skill 中找到答案，该步骤不合格。

### Phase 4：结构与效率审查

检查：

1. 主 `SKILL.md` 是否聚焦（操作步骤/判断规则内容占比是否 ≥60%）
2. 是否把次级内容放入参考文件（背景知识、理论说明、大量示例）
3. 主文件 token 估算是否合理（2K-5K tokens 为理想区间，<1K 可能太简陋，>8K 有膨胀风险）
4. 是否有重复（同一内容在多处出现）
5. 是否区分关键路径和可选路径
6. 是否有明显上下文膨胀（解释性内容超过 40%）
7. 对于**决策树型**，是否使用了渐进式披露（主文件轻量，细节在子文件按需加载）

**判断基准：** 去掉所有解释性/教学性内容后，剩余内容是否仍能独立指导 agent 完成任务。

### Phase 5：验证、安全、组合性审查

#### A. 验证与完成判定

检查：

1. 是否定义何时停止（done condition 是否明确）
2. 是否定义如何判断成功（success criteria 是否可观测）
3. 是否要求验证后再宣布完成
4. 失败情况下是否有恢复路径
5. 对于**自主式 skill**，所有完成条件是否**完全机器可验证**（不依赖人工判断）

#### B. 安全与误用防护（三层护栏模型）

**必须分三层审查，单一"注意安全"警告不等于完整防护。**

**输入层护栏**：
1. 是否对输入参数有明确验证规则
2. 是否声明不接受哪类输入（边界条件）
3. 是否避免在参数中传递敏感数据（密钥、密码、PII）

**动作前护栏**：
1. 危险操作（删除、写入、外部调用）是否有执行前确认机制
2. 不可逆操作是否要求显式人工确认
3. 是否有操作范围限制（最小权限原则）

**输出层护栏**：
1. 是否有对输出/副作用的验证
2. 高风险操作是否可回滚或有回滚说明
3. 是否有执行日志/审计要求

**高风险 skill 特别标注**：若被评审 skill 涉及密钥管理、生产环境操作、用户数据写入，必须在所有三层护栏均有覆盖时才能给安全维度 4 分以上，否则标记为 P0 风险候选项。

#### C. 可组合性

检查：

1. 是否有清晰职责边界（能否独立使用）
2. 能否作为其他 skill 的模块使用
3. 是否过度绑定个人或项目上下文
4. 输出是否可以直接被其他 skill 消费（无需额外转换）
5. 是否声明了对其他 skill 或外部服务的依赖

#### D. 自主模式特别审查（仅当执行模式含"自主式"时触发）

1. 所有完成条件是否完全机器可验证（无需人工介入）
2. 是否有**超时边界**声明（agent 不会无限循环）
3. 遇到不确定情况时是否有**降级到请求用户确认**的机制
4. 对于**接力棒型**，是否有跨 session 状态持久化机制（如文件作为接力棒）
5. 续命步骤是否被标注为 Critical/MUST（防止 agent 遗忘）

### Phase 6：输出结论

必须输出：

1. 总分与评级
2. 是否可上线
3. Skill 类型识别结论（模式 + 执行模式）
4. P0 问题
5. 十维度评分
6. 主要优点
7. 主要问题
8. 命中的反模式
9. 最小修改建议

| 结构质量 | 10 | |

## Examples

### 好 skill 示例

一个用于评审 PRD 结构完整性的 skill，如果：

- 触发明确，description 包含触发短语和时序说明
- 边界清楚，说明了不做什么
- 有固定检查步骤，带量化标准（"必须覆盖至少5个模块"）
- 有输出模板
- 有完成条件和验证机制
- 安全风险低，或有三层护栏覆盖

那么它通常是一个好 skill。

### 坏 skill 示例

一个 skill 如果大量强调"顶级专家视角""综合判断""最佳实践"，
但没有输入、步骤、输出、完成条件和验证机制，也没有量化阈值和负面指令，
它通常不是一个好 skill。

## Common Pitfalls

- 把"像专家说话"误当成"能稳定执行"
- 把所有边界情况都塞进主文件，导致主路径失焦
- skill 名称和描述太泛，导致路由混乱
- description 只描述功能，不包含触发短语和时序位置
- 只写流程，不写完成条件
- 只写建议，不写验证
- 只有"请注意安全"，没有三层护栏设计
- 声明支持自主模式，但完成条件依赖人工判断
- 试图蒸馏一个岗位的全部经验，而不是一项能力
- 不识别 skill 类型就套用统一评审标准
