# Skill 质量评审清单

按 Phase 分组，逐项快速检查。命中 P0 项即阻断，不可上线。

## Phase 0.5：Skill 类型识别（必须先完成）

- [ ] 已识别设计模式（线性流程型/决策树型/循环迭代型/接力棒型/多阶段编排型/思维框架型）
- [ ] 已识别执行模式（交互式/自主式/混合式）
- [ ] 已确认后续 Phase 的评审侧重调整方向

## Phase 1：触发层

- [ ] `name` 是否具体、可检索、低歧义
- [ ] `description` 是否同时说清做什么和什么时候用
- [ ] `description` 是否包含**触发短语**（用户可能说的话，而非纯功能描述）
- [ ] 是否说明**时序位置**（"在 X 之前/之后使用"）
- [ ] 是否包含**负向触发条件**（"不适用于 X 场景"/"什么时候不用我"）
- [ ] 是否声明**执行模式适配**（仅交互式/仅自主式/两者均支持）
- [ ] 是否容易与相邻 skill 混淆（语义重叠是否过高）
- [ ] 是否存在明显误触发或漏触发风险

> **P0 检查**：`description` 是否导致严重的触发歧义（无法区分该用还是不用）？

## Phase 2：边界与 scope

- [ ] 是否只解决一个核心问题
- [ ] 是否说明不做什么（明确排除范围）
- [ ] 是否混入多个独立职责
- [ ] 是否试图蒸馏完整岗位/完整人设/完整思维方式
- [ ] **多阶段编排型专项**：是否明确声明调用了哪些子 skill，职责边界是否清晰

> **P0 检查**：skill scope 是否本质失控（试图解决不相关的多个问题）？

## Phase 3：执行性

- [ ] 是否有清晰步骤（编号、顺序、动作动词）
- [ ] 是否有**量化阈值**（"至少3项""最多5步"，而非"足够多的""适当的"）
- [ ] 是否有**负面指令**（"不要做 X""禁止 Y"）明确列出
- [ ] 是否有**强硬语气**防止 agent 偷懒（"必须""不得""立即"而非"建议""尽量"）
- [ ] 是否有明确输入要求（格式、必需字段、前置条件）
- [ ] 是否有明确输出要求（格式、必含字段、质量标准）
- [ ] 是否有停止条件（done definition）
- [ ] 是否有分支判断（IF-THEN 规则）
- [ ] **循环迭代型专项**：是否有明确的循环退出检查清单

> **P0 检查**：主体内容是否基本不可执行（没有具体步骤、没有输入输出、没有完成条件）？

## Phase 4：结构与效率

- [ ] 主 `SKILL.md` 是否聚焦（操作步骤/判断规则占比是否 ≥60%）
- [ ] 主文件 token 估算是否合理（2K-5K tokens 为理想区间）
- [ ] 是否把背景知识/理论/大量示例放入参考文件
- [ ] 是否有重复内容（同一内容在多处出现）
- [ ] 是否区分关键路径和可选路径
- [ ] 是否有明显上下文膨胀（解释性内容超过 40%）
- [ ] **决策树型专项**：是否使用了渐进式披露（主文件轻量，细节在子文件按需加载）

## Phase 5A：验证与完成判定

- [ ] 是否定义完成条件（done condition）
- [ ] 是否定义成功标准（success criteria）
- [ ] 是否要求验证后再宣布完成
- [ ] 失败情况下是否有恢复路径
- [ ] **自主式专项**：所有完成条件是否完全机器可验证（不依赖人工判断）

> **P0 检查**：是否没有任何结束条件或验证条件？

## Phase 5B：安全与误用防护（三层护栏）

**输入层护栏**
- [ ] 是否对输入参数有明确验证规则
- [ ] 是否明确禁止传入敏感数据（密钥、密码、PII）

**动作前护栏**
- [ ] 危险操作（删除、写入、外部调用）是否有执行前确认机制
- [ ] 不可逆操作是否要求显式人工确认

**输出层护栏**
- [ ] 是否有对输出/副作用的验证
- [ ] 高风险操作是否有回滚或恢复说明

**综合判断**
- [ ] 安全防护是否分布在三层（而非全集中于一处笼统警告）
- [ ] 高风险 skill 的所有三层护栏是否均有覆盖

> **P0 检查**：高风险动作（删除/写入/外部调用/密钥/生产环境）是否无任何 guardrail，或仅有单点笼统警告？

## Phase 5C：自主模式专项（仅执行模式含"自主式"时触发）

- [ ] 所有完成条件是否完全机器可验证
- [ ] 是否有超时边界声明
- [ ] 遇到不确定情况是否有降级到请求用户确认的机制
- [ ] **接力棒型专项**：是否有跨 session 状态持久化机制
- [ ] **接力棒型专项**：续命步骤是否被标注为 Critical/MUST

> **P0 检查**：声明为自主模式，但完成条件核心依赖用户实时干预？

## Phase 6：输出

- [ ] 是否输出了 Skill 类型识别结论（模式 + 执行模式）
- [ ] 是否输出了总分与评级
- [ ] 是否输出了 P0 问题列表
- [ ] 是否输出了十维度评分
- [ ] 是否输出了主要优点、问题、反模式、修改建议
- [ ] 不同评审者使用该 skill 能否输出格式一致的报告
