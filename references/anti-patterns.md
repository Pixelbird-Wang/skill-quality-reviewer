# Skill 常见反模式

每个反模式包含：特征描述、识别方法、典型表现。

## 1. 人设伪装能力

**特征：** skill 大量使用"你是顶级专家""你拥有丰富经验""你具备深刻洞察"等身份描述，但不转化为任何具体动作。

**识别方法：** 如果去掉所有人设描述后，skill 的正文变得空洞或不可执行，即命中此反模式。

**典型表现：** "你是一位资深产品经理，具备敏锐的市场洞察力"后面没有跟任何分析步骤或判断标准。

## 2. 全面但不可执行

**特征：** 覆盖了几乎所有可能情况，但每条都缺乏具体操作步骤、判断标准或输出格式。

**识别方法：** 对任意一条建议问"具体怎么做"，如果无法在 skill 中找到答案，即命中。

**典型表现：** "根据场景灵活调整策略""综合运用多种方法""结合实际情况判断"。

## 3. 工具描述歧义

**特征：** 对工具的使用描述模糊，agent 无法判断何时该用、何时不该用，或使用时需要传递什么参数。

**识别方法：** 如果 tool 名称或使用场景的描述可以让 agent 产生两种以上的理解，即命中。

**典型表现：** 工具描述只说"可以查询数据"，但没说查什么数据、何时查、返回后怎么处理。

## 4. 工作流蔓延

**特征：** 从一个核心问题出发，逐步扩展到多个不相关的子任务，最终 scope 失控。

**识别方法：** 检查 skill 的步骤，如果中间步骤引入了与主问题不直接相关的新任务，即命中。

**典型表现：** 评审 PRD 的 skill，中途要求"顺便检查一下竞品分析是否完整"、"建议补充用户画像调研"。

## 5. 无验证真空

**特征：** 描述了完整的流程和输出，但没有定义何时停止、如何判断成功、失败后怎么处理。

**识别方法：** 看完 skill 后如果不知道"做完的标志是什么"，即命中。

**典型表现：** "输出分析报告"但没有说明报告应包含哪些必须项、达到什么质量才算完成。

## 6. 上下文膨胀

**特征：** 在核心文件中塞入大量背景知识、理论说明、最佳实践，导致关键路径被稀释。

**识别方法：** 如果 SKILL.md 中超过 40% 的内容是解释性/教学性文字而非操作步骤或判断规则，即命中。

**典型表现：** 主文件里有大段的行业分析、理论框架、历史背景，但操作步骤只有寥寥数行。

## 7. 样例绑架

**特征：** 用大量具体案例替代抽象原则，agent 遇到非样例场景时无法泛化。

**识别方法：** 如果去掉所有示例后，skill 失去了可执行性，即命中。

**典型表现：** 列了20个场景的应对方式，但没有总结出通用的判断逻辑，遇到第21个场景时 agent 不知道该怎么做。

## 8. 硬编码隐性判断

**特征：** 把需要经验、审美、直觉才能做出的判断硬编码成规则，或者试图蒸馏无法流程化的能力。

**识别方法：** 检查 skill 中是否有需要"品味""感觉""经验判断"才能执行的指令，即命中。

**典型表现：** "如果方案看起来不够专业，请调整语气""像资深专家那样评估这个方案的可行性"。

## 9. 自主模式歧义

**特征：** skill 声明可在自主模式下运行（无需用户实时干预），但其完成条件、分支判断或错误处理实际依赖用户实时反馈，导致自主执行时要么卡死、要么产生幻觉性"完成"。

**识别方法：** 在 skill 中搜索"询问用户""请用户确认""等待用户输入""请告知"等表述。如果这些步骤出现在主路径中（而非降级/异常路径），且该 skill 声称支持自主运行，即命中。

**严重程度：** 高。会导致 agent 在自主执行时产生不可预期行为，或在交互无响应时无限等待。

**典型表现：**
- skill 描述说"支持自主执行"，但核心步骤包含"完成后请告诉用户是否继续"
- 完成条件定义为"直到用户满意"——这在自主模式下完全不可验证
- 分支判断依赖"根据用户反应决定"，没有机器可验证的代替判断

**修复方向：** 为每个"依赖用户"的判断点提供机器可验证的替代条件；或明确声明该 skill 仅支持交互式执行。

## 10. 护栏层缺失

**特征：** skill 涉及高风险操作（删除、写入、外部调用、密钥处理、生产环境操作），但所有防护集中在一处笼统的安全提醒，未按"输入验证→执行前确认→输出验证"三层分别设计护栏。

**识别方法：** 检查 skill 中的所有安全相关描述是否集中于单一位置，且使用了"请谨慎操作""注意安全""确认后执行"等模糊措辞而非明确的操作前/操作中/操作后具体检查。如果找不到三层护栏结构，即命中。

**严重程度：** 高。单层防护在 agent 执行路径稍偏移时即完全失效；三层护栏在单层被绕过时仍有保护。

**典型表现：**
- "执行删除前请谨慎确认"——一句话涵盖全部安全防护
- 有"注意密钥安全"但没有明确禁止将密钥写入参数或日志
- 写入操作没有回滚说明，也没有操作后验证步骤

**修复方向：** 将安全防护拆分为三层：
1. **输入层**：验证参数合法性，明确禁止哪类输入（特别是敏感数据）
2. **动作前层**：危险操作执行前的确认机制，不可逆操作要求显式人工确认
3. **输出层**：操作后验证副作用是否符合预期，提供回滚或恢复路径
